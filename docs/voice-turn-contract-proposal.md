# Voice-turn contract proposal

**Status:** Proposal — needs backend team review + sign-off before Phase 3 wires the route
**Owner:** Voice service author + orchet-backend orchestrator owner
**Implements:** the safety boundary mandated by VOICE-ARCHITECTURE-1 v5 ("voice service does NOT execute tools; orchestrator owns tool policy")
**Status of routes:** PROPOSED. Neither `/voice/turn` nor `/voice/confirm-action` exists on `api.orchet.ai` yet. They land in Phase 3.

---

## The contract in one paragraph

The voice service receives audio over WebRTC, runs the streaming STT → LLM → TTS pipeline, and when the LLM emits a function-call frame, POSTs that function-call payload to `api.orchet.ai/voice/turn`. The backend gateway routes the request internally to the orchestrator. The orchestrator decides one of three outcomes — `executed`, `requires_visual_confirmation`, or `denied` — and returns the decision to voice. Voice never executes a tool directly. The client confirms high-risk actions visually via a separate `POST /voice/confirm-action` call. All decisions and confirmations are audited backend-side.

---

## Route 1 — `POST /voice/turn`

**Purpose:** Voice service asks the orchestrator to decide how to handle an LLM-proposed tool call.

**Caller:** orchet-voice service (Pipecat pipeline, when LLM emits a function-call frame).

**Auth:** `Authorization: Bearer <ORCHET_INTERNAL_TOKEN>` (the same internal service-to-service token used between gateway and backend). The user's identity is carried in the request body, not the JWT — voice validated the user JWT at WebRTC connection-open and stamps `user_id` here.

### Request

```http
POST /voice/turn
Host: api.orchet.ai
Authorization: Bearer <ORCHET_INTERNAL_TOKEN>
Content-Type: application/json
Idempotency-Key: <uuid v7 generated by voice service per turn>
```

```json
{
  "session_id": "voice_01HK3PW5J7M4ZGYB6X9T2QD8VN",
  "user_id": "54c86c7d-c44e-4dbd-83d3-40965e970490",
  "channel": "voice",
  "locale": "en-US",
  "agent_id": "lumo-rentals-trip-planner",
  "turn_index": 4,
  "transcript_partial": "book me a flight to tokyo tomorrow morning",
  "tool_call": {
    "name": "book_flight",
    "arguments": {
      "origin": "SFO",
      "destination": "NRT",
      "departure_date": "2026-05-11",
      "departure_time_window": "morning"
    }
  },
  "context": {
    "device_kind": "web",
    "client_ip_region": "us-east",
    "voice_latency_ms_so_far": 720
  }
}
```

### Response — outcome 1: `executed`

The orchestrator ran the tool itself (low-risk: search, lookup, compare, read-only). Returns the result so voice can keep the conversation flowing.

```json
{
  "outcome": "executed",
  "tool_call_id": "tc_01HK3PW6...",
  "result": {
    "flights": [
      { "carrier": "JAL", "depart": "2026-05-11T09:30:00Z", "price_usd": 850 },
      { "carrier": "ANA", "depart": "2026-05-11T10:15:00Z", "price_usd": 890 }
    ]
  },
  "voice_message_hint": "I found two morning flights. JAL at 9:30 for $850 or ANA at 10:15 for $890. Which would you prefer?"
}
```

`voice_message_hint` is optional — when present, voice prefers to speak this exact phrasing (consistent voice across channels). When absent, voice continues the LLM stream with the tool result injected as a tool-result message.

### Response — outcome 2: `requires_visual_confirmation`

The orchestrator decided this tool call is high-risk and needs an explicit user confirmation tap. Voice does NOT execute. Voice tells the user via TTS and emits a WebRTC data-channel event so the client can render a confirmation modal.

```json
{
  "outcome": "requires_visual_confirmation",
  "confirmation_id": "conf_01HK3PW7...",
  "voice_message": "I've prepared a JAL flight from SFO to NRT tomorrow at 9:30 AM for $850. Please confirm on screen.",
  "confirmation_payload": {
    "title": "Confirm booking",
    "summary": "JAL · SFO → NRT · 2026-05-11 09:30 · $850",
    "details": [
      { "label": "Departure", "value": "May 11, 9:30 AM (SFO)" },
      { "label": "Arrival", "value": "May 12, 1:15 PM (NRT)" },
      { "label": "Price", "value": "$850 (1 adult, economy)" },
      { "label": "Cancellation", "value": "Refundable until 24h before departure" }
    ],
    "confirm_action": "book_flight",
    "expires_at": "2026-05-10T18:35:00Z"
  }
}
```

Voice speaks `voice_message` and forwards `confirmation_payload` over the WebRTC data channel. Client renders modal. When user taps "Confirm", client calls `POST /voice/confirm-action` (see route 2). When user taps "Cancel" or modal expires, client calls `POST /voice/confirm-action` with `accepted: false`.

### Response — outcome 3: `denied`

The orchestrator's policy refuses the tool call entirely for voice channel — e.g., voice is not allowed to trigger payments above a threshold, or the user's account is in a state that blocks this tool. Voice tells the user via TTS.

```json
{
  "outcome": "denied",
  "reason_code": "voice_not_allowed_for_payment",
  "voice_message": "I can't process payments through voice mode. Please open the booking in the app to complete this."
}
```

### Errors

- `401` — invalid or missing internal token
- `403` — `user_id` doesn't match a valid session
- `404` — `agent_id` or tool name unknown
- `409` — duplicate `Idempotency-Key` (orchestrator returns the original decision)
- `429` — rate limited
- `5xx` — orchestrator error; voice service tells user "I'm having trouble — try again in a moment" via TTS

## Route 2 — `POST /voice/confirm-action`

**Purpose:** Client submits the user's confirmation tap (or rejection) for a `requires_visual_confirmation` outcome. The orchestrator commits the tool call if accepted.

**Caller:** orchet-web or orchet-ios client (NOT the voice service — voice doesn't see the user's tap).

**Auth:** `Authorization: Bearer <supabase access_token>` (the user's JWT, the same one used for all other client → gateway calls).

### Request

```http
POST /voice/confirm-action
Host: api.orchet.ai
Authorization: Bearer <supabase access_token>
Content-Type: application/json
```

```json
{
  "session_id": "voice_01HK3PW5J7M4ZGYB6X9T2QD8VN",
  "confirmation_id": "conf_01HK3PW7...",
  "accepted": true
}
```

### Response — accepted

```json
{
  "result": "executed",
  "tool_call_id": "tc_01HK3PWA...",
  "summary": "Flight booked. Confirmation code JAL-XYZ123. Email sent.",
  "voice_continuation_hint": "Your JAL flight is confirmed. Confirmation code JAL-XYZ123. I've emailed the details."
}
```

The orchestrator also emits a server-sent event back to the voice service (out-of-band via a separate WebSocket or polling) so voice can speak the `voice_continuation_hint`.

### Response — rejected / expired

```json
{
  "result": "cancelled",
  "voice_continuation_hint": "No problem, I've cancelled that. What would you like to do instead?"
}
```

## High-risk action policy

The tool registry (in `orchet-backend`) is extended with two new flags per tool:

```yaml
tools:
  - id: search_flights
    voice_allowed: true
    requires_visual_confirmation: false
    risk_level: low

  - id: book_flight
    voice_allowed: true
    requires_visual_confirmation: true
    risk_level: high
    voice_message_pre_confirm: "I've prepared your flight booking. Please confirm on screen."

  - id: charge_card
    voice_allowed: false
    requires_visual_confirmation: true
    risk_level: critical
    voice_denial_message: "Payments must be confirmed in the app. Please open the booking to pay."
```

### Default risk classification

| Risk | Examples | Voice behavior |
|---|---|---|
| Low | Search, lookup, compare, read-only memory, get-status | `executed` |
| Medium | Send message, set reminder, draft (not send) | `executed` with `voice_message_hint` for clarity |
| High | Book, schedule, modify booking, share contact | `requires_visual_confirmation` |
| Critical | Charge card, transfer money, sign contract, delete account, irreversible legal/compliance | `denied` for voice — must use visual app |

The classification lives in the tool manifest, not in voice code. Voice is provider-agnostic; it only knows the three outcomes.

## Audit logging (backend-side)

Every `/voice/turn` and `/voice/confirm-action` call writes to the existing audit log table with:

- `session_id`, `user_id`, `agent_id`, `tool_call_name`, `outcome`, `reason_code`
- `requested_at`, `decided_at`, `confirmed_at` (for visual-confirmation flow)
- The raw request payload (with PII redacted per existing policy)

Voice service never writes to audit. Single source of truth.

---

## Idempotency contract

Voice generates a UUIDv7 `Idempotency-Key` per turn and includes it on `/voice/turn`. If voice retries (network blip, container restart), the orchestrator returns the previously-cached decision rather than re-running the tool. Cache TTL: 24 hours.

`/voice/confirm-action` is idempotent on `confirmation_id` — clients can safely retry the user's tap.

---

## What voice service must NOT do

Stated for the record:

- Voice service must NOT execute tools directly — no `POST /tools/{id}` from voice, ever
- Voice service must NOT decide confirmation requirements — that's orchestrator policy
- Voice service must NOT cache tool results client-side — orchestrator owns state
- Voice service must NOT bypass orchestrator on retries — same auth path, same audit
- Voice service must NOT mint or modify user tokens — Bearer JWT is validated, never re-issued

## Open questions for backend team

1. **Existing `/orchestrator/turn` route reuse?** Text chat uses it today. Should `/voice/turn` be a sibling on the gateway that proxies to a new orchestrator route, or should it map to `/orchestrator/turn` with `channel: "voice"`? Recommend: separate gateway route + separate orchestrator route, but share the business logic underneath. Keeps voice-specific policy testable in isolation.
2. **Streaming response for `/voice/turn` `executed` outcome?** Today the response is a single JSON blob. If tool results are large (search returning 50 flights), should this be SSE? Recommend: SSE for `executed` only, fixed JSON for `requires_visual_confirmation` and `denied`.
3. **Where does `voice_continuation_hint` get delivered to voice?** Three options: (a) SSE back to voice on the same TCP connection; (b) separate WebSocket from orchestrator to voice; (c) voice polls. Recommend: (b) is cleanest — voice subscribes once per session, receives confirmation events.
4. **Idempotency cache implementation?** Redis on backend side, keyed on `(user_id, idempotency_key)`. 24h TTL.

## Phase 1 readiness gate

This contract is a **proposal**, not a built route. Phase 1 (orchet-voice skeleton) does not need `/voice/turn` to exist — Phase 1 ships echo round-trip only. The contract gets refined during Phase 2 (when voice has a real LLM and the function-call frames start appearing) and the route lands at the start of Phase 3.

Backend team review of this contract should happen before Phase 3 starts (~3 weeks out from Phase 0 start). Earlier review welcome.
